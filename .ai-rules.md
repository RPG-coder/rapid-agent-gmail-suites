# 🤖 Gemini-CLI Workspace Rulebook: Rapid Agent Gmail Suite

This workspace contains a specialized, event-driven multi-agent Google Workspace Add-on built for the Google Rapid Agent Hackathon. All AI generations must conform to these architectural guardrails.

---

## 📐 1. Global Architectural Constraints

- **Language:** Python 3.11+ for all agent business logic and data streaming.
- **Framework:** Google Agent Development Kit (ADK) paired with standard Model Context Protocol (MCP).
- **UI Delivery:** Google Workspace Card Service (Gmail Sidebar UI via Apps Script in `publish/sidebar-ui/`).
- **Deployment Paradigm:** Serverless containers (Google Cloud Run) orchestrated via Pub/Sub and EventArc triggers.
- **Visibility:** Unlisted Google Workspace Marketplace deployment.

---

## 📂 2. Directory Boundaries & Code Ownership

You must isolate file generation to their respective boundaries. Never mix domain logic.

### 🔹 `agents/smart_email_manager/`

- **Responsibility:** Real-time email categorization, automated drafts, and semantic bucket grouping.
- **Stack:** Voyage AI (embeddings) + MongoDB (operational vector cluster/change streams).
- **Triggers:** Event-driven via Google Cloud Pub/Sub (`pubsub_handler.py`). High-volume asynchronous cleanups pass to `cloud_run_worker.py`.

### 🔹 `agents/inbox_analytics/`

- **Responsibility:** Analytical RAG, computing email velocity patterns, and response metrics.
- **Stack:** Fivetran pipelines (offline transform) ➡️ BigQuery (Data Warehouse) 🔄 Vertex AI Vector Search.
- **Querying:** Direct SQL interaction via `bigquery_client.py` and semantic data parsing via `vertex_vector.py`.

### 🔹 `agents/self_arizeing_manager/`

- **Responsibility:** Full-stack LLM observability, trace diagnostics, and automated prompt performance tuning.
- **Stack:** OpenInference Tracing SDK (`openinference-instrumentation-google-genai`) ➡️ Arize Phoenix Server / Phoenix Cloud.
- **Loop:** Uses Gemini 3 as a LLM-as-a-Judge to evaluate trace logs, then updates `llm_semantic_note` fields in MongoDB via `MongoDB MCP`.

### 🔹 `shared/`

- **Responsibility:** Single Source of Truth for schemas and database connections.
- **Rules:** Custom MCP server JSON representations reside in `shared/mcp_schemas/`. Common utility clients go in `shared/utils/`.

### 🔹 `publish/sidebar-ui/`

- **Responsibility:** Gmail Sidebar UI components built with Google Apps Script Card Service.
- **Rules:** 
    - Use the `create[Name]Card()` function naming convention for card builders.
    - Separate UI layout logic from Action Handlers.
    - **State Management:** Apps Script is stateless. Pass all required state (active IDs, search queries, history) through `Action.setParameters()`.
    - **Naming Convention:** Prefix or suffix action functions to avoid global namespace collisions (e.g., `on[Action]SEM`, `on[Action]SAM`, `on[Action]Analytics`).

## 🛠️ 3. Execution & Code Generation Guardrails

When generating Python code or configuration setups, strictly implement these design patterns:

### A. Model Context Protocol (MCP) Compliance

- Always format tool parameters matching the exact schema definitions found in `shared/mcp_schemas/`.
- Wrap all MCP network and tool communication within asynchronous (`async/await`) handlers to maximize high-throughput event processing.

### B. Invisible Telemetry

- Core business logic in `smart_email_manager` and `inbox_analytics` must remain clean and free from performance-monitoring pollution.
- All tracing, tracing providers, and observability hook initializations must be abstracted away inside `agents/self_arizeing_manager/telemetry.py`.

### C. State Mutation Isolation

- Agents must not modify MongoDB documents or configurations outside their scope unless invoking an authorized MCP tool call.
- Keep data streams unidirectional where possible: Gmail Event ➡️ Pub/Sub ➡️ Python ADK Agent ➡️ MCP Action Writeback.

### D. UI Styling Standards

- Consistently use the following color palette for `CardService` elements:
    - **Success / Active**: `#0F9D58` (Green)
    - **Warning / Inactive / Hint**: `#999999` (Grey)
    - **Danger / Alert / Error**: `#DB4437` (Red)
    - **Primary Text**: `#1A1A1A`
    - **Secondary/Italic Text**: `#333333`
